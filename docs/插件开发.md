## 1. 概述

插件是独立的 Vue 组件，通过 `manifest.json` 声明元数据，在运行时由主程序加载。主程序与插件解耦，插件需预构建为 UMD 格式的 `App.js`，并依赖主程序提供的 Vue、Element Plus、Icon。

## 2. 目录结构

```
plugins-ext/
└── 端口查杀/              # 插件目录名（建议与插件标题一致）
    ├── manifest.json     # 必填，插件元数据
    ├── App.vue           # 必填，插件入口组件
    ├── App.js            # 构建产物，由 build:plugins 生成
    └── style.css         # 构建产物，由 build:plugins 生成
```

### 2.1 插件目录与路径

| 环境 | 插件目录 |
|------|----------|
| 开发 | `./plugins`（由 `build:plugins` 从 `plugins-ext/` 构建输出） |
| 打包后 | `~/.ly/tools/plugins`（用户目录） |

源码在 `plugins-ext/`，执行 `npm run build:plugins` 会构建到 `plugins/` 并生成各插件的 zip 包；主程序从 `plugins/` 加载。

## 3. manifest.json

### 3.1 必填字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | string | 插件唯一标识，如 `port-killer` |
| `route` | string | 路由路径，如 `/port-killer` |
| `icon` | string | 图标，如 `ri:terminal-box-line`（Iconify 格式） |
| `description` | string | 插件描述 |
| `color` | string | 卡片渐变色，如 `linear-gradient(135deg, #667eea 0%, #764ba2 100%)` |
| `supportedOS` | string[] | 支持的系统，如 `["win64"]`、`["win64","linux"]` |

### 3.2 可选字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `title` | string | 显示名称，缺省时使用目录名 |
| `version` | string | 版本号 |
| `fullScreen` | boolean | 是否全屏显示 |
| `electronAction` | string | 点击卡片时调用的 electronAPI 方法名（如 `openLightOffWindow`） |
| `electronActions` | string[] | 点击卡片时依次调用的多个 electronAPI 方法名，可与 `electronAction` 共存 |
| `minLyToolsVersion` | string | 最低主程序版本（semver）；不满足则插件不加载。依赖 `electronAPI.plugin.invokeMain` 等较新能力的插件应写明，例如 `1.1.0` |

当 `fullScreen` 为 true 且配置了 `electronAction`/`electronActions` 时，主程序会在点击卡片时调用对应方法（若 electronAPI 中存在），否则走默认路由跳转。`electronAction` 先执行，再执行 `electronActions` 中各方法。

### 3.3 supportedOS 取值

- `win64` / `win32` / `windows` → Windows
- `linux` → Linux
- `darwin` / `macos` → macOS

### 3.4 完整示例（含所有属性）

```json
{
  "id": "my-plugin",
  "route": "/my-plugin",
  "title": "我的插件",
  "icon": "ri:terminal-box-line",
  "description": "插件功能描述，展示在首页卡片上",
  "color": "linear-gradient(135deg, #667eea 0%, #764ba2 100%)",
  "version": "1.0.0",
  "supportedOS": ["win32", "linux", "darwin"],
  "fullScreen": true,
  "electronAction": "openLightOffWindow",
  "electronActions": ["closeLockWindow"],
  "minLyToolsVersion": "1.0.0"
}
```

> `fullScreen` 为 `true` 且配置了 `electronAction`/`electronActions` 时，点击卡片会调用对应 electronAPI 方法；否则走默认路由跳转。普通插件只需省略 `fullScreen`、`electronAction`、`electronActions` 即可。

| 属性 | 必填 | 说明 |
|------|:----:|------|
| `id` | ✓ | 插件唯一标识，用于路由、独立窗口 `?pluginId=xxx` |
| `route` | ✓ | 路由路径，如 `/port-killer` |
| `icon` | ✓ | 图标（Iconify 格式，如 `ri:terminal-box-line`） |
| `description` | ✓ | 插件描述 |
| `color` | ✓ | 卡片渐变色 |
| `supportedOS` | ✓ | 支持系统：`win32`/`win64`/`windows`、`linux`、`darwin`/`macos` |
| `title` | | 显示名称，缺省时使用插件目录名 |
| `version` | | 版本号，展示在卡片上 |
| `fullScreen` | | 是否全屏，默认 `false` |
| `electronAction` | | 点击卡片时调用的 electronAPI 方法名 |
| `electronActions` | | 点击卡片时依次调用的多个 electronAPI 方法名 |
| `minLyToolsVersion` | | 最低主程序版本，不满足则插件不加载 |

**注意**：`pluginDir` 由主程序在加载时注入（插件目录名），无需在 manifest 中声明。

### 3.5 `electronAction` / `electronActions` 与 `electronAPI.plugin.invokeMain` 的区别

二者都用于「插件 + Electron」，但**职责不同**：前者是 **manifest 声明的首页卡片行为**（调用主程序已固定的无参 API）；后者是 **插件页面内通过 `electronAPI.plugin.invokeMain` 调用插件目录里的 Node 脚本**（可传参、业务在插件包内）。不要混用场景。

**对框架的影响**：主程序只在 `preload` 中增加 **`electronAPI.plugin` 命名空间**（内含 `invokeMain` 一处），**不因单个插件再增加顶层 API**；IPC 仅 **`invoke-plugin-main`** 一条。新增类似「数据库同步」这类带主进程脚本的插件时，**无需再改 preload / 主进程**，只在插件目录内增加脚本与（可选）`package.json` 即可。

#### 总览

| 对比项 | `electronAction` / `electronActions`（manifest） | `electronAPI.plugin.invokeMain` |
|--------|--------------------------------------------------|----------------------------------|
| 配置位置 | `manifest.json` 可选字段 | 无 manifest 字段；在插件 `App.vue`（或逻辑）里代码调用 |
| 主程序 IPC | 不新增；复用已有 `electronAPI` 方法名 | 固定一条通用通道 `invoke-plugin-main` |
| 与插件目录内脚本 | **无关** | **有关**；主进程按 `pluginId` 定位目录，加载指定 `.js` 并调用其入口方法 |
| 典型用途 | 全屏/独立窗口、与内置窗口联动（关灯、关锁屏等） | 需在主进程执行的逻辑（数据库、本地文件等），由插件自带脚本实现 |

#### 触发时机与调用形态

| 对比项 | `electronAction` / `electronActions` | `plugin.invokeMain` |
|--------|--------------------------------------|----------------------|
| 何时触发 | 用户在**首页点击插件卡片**时 | 插件界面内任意时机（如按钮点击） |
| 前置条件 | 需 `fullScreen: true`，且方法名在 `electronAPI` 上存在 | 需在 Electron 中；插件已构建到 `plugins/` 且存在对应脚本 |
| 调用写法 | 主程序执行 `electronAPI['方法名']()`，**无参数** | `electronAPI.plugin.invokeMain(pluginId, script, method, args)`，**可传 `args`** |
| 能否传业务数据 | **不能**（无法传连接串、路径等） | **能**（`args` 一般为对象） |
| 不配置时的行为 | 未配或调用失败时，回退为 `router.push(route)` 进入插件页 | 不涉及首页卡片；仅在代码显式调用时执行 |

#### 扩展主程序的方式

| 对比项 | `electronAction` / `electronActions` | `plugin.invokeMain` |
|--------|--------------------------------------|----------------------|
| 新增能力是否改 preload | 若要用**新**动作，通常需在 `preload.js` **增加**对应 `electronAPI` 方法（属主程序扩展） | **不需要**为每个插件加专用 IPC；**仅使用已有** `plugin.invokeMain` |
| 业务代码位置 | 主进程里与该方法绑定的逻辑（如 `openLightOffWindow` 在主进程实现） | **插件目录内** `.js`（如 `helper.js`），主进程只做安全转发 |

#### 选型建议（简要）

| 需求 | 更合适的方式 |
|------|----------------|
| 点首页卡片只想打开某个**已有**全屏/窗口能力 | `fullScreen` + `electronAction` / `electronActions` |
| 点首页卡片进入插件页，再在页内操作 | 不配 `electronAction`，走默认路由；页内用 `plugin.invokeMain` |
| 主进程执行带参数任务（库同步、处理文件等） | **`electronAPI.plugin.invokeMain`**，脚本放在插件目录 |

## 4. App.vue

### 4.1 基本结构

插件入口为单文件组件 `App.vue`，使用 Vue 3 Composition API、Element Plus、@iconify/vue。与主程序用法一致，无需额外配置。

```vue
<template>
  <div class="my-plugin">
    <h2>我的插件</h2>
    <!-- 使用 Element Plus 组件 -->
    <el-button type="primary" @click="handleClick">点击</el-button>
  </div>
</template>

<script setup>
import { ref } from 'vue'
import { ElMessage } from 'element-plus'
import { Icon } from '@iconify/vue'

const count = ref(0)
function handleClick() {
  count.value++
  ElMessage.success(`点击了 ${count.value} 次`)
}
</script>

<style scoped>
.my-plugin { padding: 20px; }
</style>
```

### 4.2 独立窗口模式

插件可通过 `?pluginId=xxx` 以独立窗口打开。若需区分主窗口和独立窗口，可接收 `standalone` 属性：

```vue
<script setup>
const props = defineProps({
  standalone: { type: Boolean, default: false }
})
// standalone 为 true 时表示在独立窗口运行
</script>
```

### 4.3 深色模式

主框架支持浅色/深色模式切换，切换时会在 `document.documentElement` 上添加或移除 `dark` 类。插件可跟随变化：

1. **Element Plus 组件**：主程序已加载 Element Plus 深色样式，`el-button`、`el-card`、`el-input` 等会自动适配，无需额外处理。
2. **自定义样式**：若插件有硬编码的浅色（如 `#f0f0f0`、`#909399`），需增加 `html.dark` 选择器覆盖：

```css
/* 浅色默认 */
.log-item { border-bottom: 1px solid #f0f0f0; }
.log-time { color: #909399; }

/* 跟随主框架深色模式 */
html.dark .log-item { border-bottom-color: #30363d; }
html.dark .log-time { color: #8b949e; }
```

3. **注入 theme**：主程序通过 `provide('theme', theme)` 提供当前主题 ref，插件可用 `inject('theme')` 获取并监听变化，用于需要响应式的逻辑。

### 5.2 插件主进程脚本（通用调用，推荐）

与 manifest 里 `electronAction` 的差异见 **§3.5**（表格对照）。

若需在主进程执行 Node 能力（如访问数据库、文件系统），**不要**为主程序增加专用 IPC；将逻辑放在插件目录下的 `.js` 文件中（与 `App.vue` 同级，构建时一并复制到 `plugins/`），通过 **`electronAPI.plugin.invokeMain`** 调用。

| 参数 | 说明 |
|------|------|
| `pluginId` | `manifest.json` 的 `id` |
| `script` | 脚本文件名，如 `helper` 或 `helper.js`（仅允许字母、数字、`.`、`_`、`-`） |
| `method` | 该脚本在 `module.exports` 上暴露的方法名（技术含义，非「导出数据」） |
| `args` | 传入该方法的第一个参数（一般为对象） |

主进程会从 `plugins/<插件目录>/` 加载对应脚本并执行约定方法（与「数据导出」等业务语义无关，仅指 Node 模块的调用入口）；**业务代码仅存在于插件包内**，主框架只保留 **`invoke-plugin-main` 这一条**通用 IPC 与 **`electronAPI.plugin.invokeMain`** 一处预加载封装。

**示例**：带主进程脚本与插件内 `package.json` 的完整示例为「数据库同步」插件，其专用开发说明见 **`plugins-ext/数据库同步/开发说明.md`**（构建后位于 `plugins/数据库同步/开发说明.md`，可随 zip 分发）。

#### 插件主进程脚本的 npm 依赖（与框架解耦）

若脚本需要第三方包（如 `mysql2`、`pg`），**不要**加到主程序根目录的 `package.json`。在插件目录（`plugins-ext/<插件名>/`）下增加 **`package.json`**，声明 `dependencies`；执行 **`npm run build:plugins`** 时会在输出目录 `plugins/<插件名>/` 自动执行 **`npm install --omit=dev`**，将依赖安装到 **`plugins/<插件名>/node_modules`**。

主进程通过 Node 的 **`createRequire(插件目录/package.json)`** 加载脚本，脚本内的 `require('第三方包')` 从**该插件**的 `node_modules` 解析，**不占用框架依赖**，也不会与主程序其它依赖冲突。

| 位置 | 作用 |
|------|------|
| `plugins-ext/<插件名>/package.json` | 源码侧声明插件专用依赖（随仓库或插件 zip 分发） |
| `plugins/<插件名>/package.json` + `node_modules/` | 构建产物；运行时主进程从此解析 |
| 无 `package.json` 的插件 | 仍可用 `createRequire(脚本路径)` 加载；仅当脚本内 `require` 第三方包时需在插件目录提供 `package.json` 并安装依赖 |

渲染进程若需 **回调风格**（不污染 async/await 与框架），可在插件内自行封装，例如：

```js
function invokeMainWithCallback(pluginId, script, method, args, callback) {
  const api = window.electronAPI
  if (!api?.plugin?.invokeMain) {
    callback(new Error('需在 Electron 中使用'), null)
    return
  }
  api.plugin.invokeMain(pluginId, script, method, args)
    .then((res) => callback(null, res))
    .catch((err) => callback(err, null))
}

invokeMainWithCallback('my-plugin-id', 'helper', 'run', { foo: 1 }, (err, res) => {
  if (err) { /* ... */ return }
  // res 为脚本方法返回值
})
```

脚本示例（`helper.js`）：

```js
module.exports = {
  async run(payload) {
    return { success: true, message: 'ok' }
  },
}
```

### 5.3 扩展能力可支撑的应用（举例）

在 **`electronAPI.plugin.invokeMain`** + 插件目录 **`package.json` / `node_modules`** + 主进程脚本（`.js`）的前提下，**不必再改主程序 preload**，即可在单进程内扩展下列类型能力（插件 UI 仍为 `App.vue`，重逻辑放脚本）：

| 方向 | 示例插件设想 | 依赖/说明 |
|------|----------------|-----------|
| 数据与存储 | 除数据库同步外：**Redis / MongoDB 连接测试**、**本机 SQLite 查询与导出**、**大文件 CSV ↔ 表导入导出** | 对应官方 Node 驱动装在插件 `package.json` |
| 文件与系统 | **批量重命名 / 按规则整理文件夹**、**校验和与重复文件查找**、**日志 tail 与高亮**（读本地路径） | `fs`、`path`、可选 `chokidar` 等仅在插件内 |
| 文本与格式 | **JSON/YAML/TOML 格式化与校验**、**Markdown/HTML 预览**（主进程读文件再回传）、**简单模板引擎批量生成文件** | 纯 JS 包即可 |
| 网络与接口 | **REST / GraphQL 调试器**（主进程 `fetch`/`http` 避免 CORS）、**Webhook 接收与展示**（需额外 IPC 时可再评估）、**本机服务健康检查** | 敏感逻辑仍在插件脚本，不扩大主程序 API 面 |
| 开发运维 | **Git 状态可视化**（调用 `simple-git` 等）、**端口批量探测**（与现有端口查杀互补）、**环境变量对比** | 命令行工具可用 `child_process` 在脚本内执行 |
| 安全与隐私 | **字符串加密/哈希**（本地计算不上传）、**证书或密钥文件查看** | 注意不在日志中打印密钥 |

**边界**：不适合把「主窗口生命周期、全局快捷键、系统托盘」等仍应放在主程序；**长时间阻塞**或**常驻子进程**需在插件脚本内自行控制，避免卡死主进程。

以上均为**产品方向举例**；本仓库已落地的参考实现为 **「数据库同步」** 插件（见 `plugins-ext/数据库同步/开发说明.md`）。

## 6. 运行时加载机制

### 6.1 自定义协议 ly-plugin://

主进程注册 `ly-plugin://` 协议，从插件目录提供静态文件。完整定义见 [ly-plugin协议定义.md](./ly-plugin协议定义.md)。

**URL 格式**：`ly-plugin://plugin/<encodeURIComponent(pluginDir)>/<file>`

| 示例 | 说明 |
|------|------|
| `ly-plugin://plugin/%E7%AB%AF%E5%8F%A3%E6%9F%A5%E6%9D%80/App.js` | 插件主入口（UMD） |
| `ly-plugin://plugin/%E7%AB%AF%E5%8F%A3%E6%9F%A5%E6%9D%80/style.css` | 插件样式 |

> **说明**：pluginDir 必须放在 path 中，避免 host 编码解析问题导致加载失败。

**协议处理流程**（主进程 `protocol.handle`）：

```
请求 URL: ly-plugin://plugin/<encoded>/App.js
    │
    ▼
解析 pathname → parts = [encoded, "App.js"]
    │
    ▼
pluginDir = decodeURIComponent(parts[0])
file = parts.slice(1).join("/")
    │
    ▼
filePath = pluginsRoot/pluginDir/file
    │
    ▼
校验：filePath 在 pluginsRoot 内 且 文件存在
    │
    ▼
读取文件 → 按扩展名设置 MIME → 返回 Response
```

**MIME 映射**：`.js` → `application/javascript`，`.css` → `text/css`，`.json` → `application/json`

### 6.2 加载流程

1. 主进程 `getPluginList()` 扫描插件目录，读取各插件的 `manifest.json`
2. 路由根据插件列表动态注册 `path: manifest.route`
3. 访问插件路由时，`getPluginComponentLoader(pluginDir)` 被调用
4. 渲染进程 IPC 调用 `getPluginEntryUrl(pluginDir)` 获取 `ly-plugin://plugin/encoded/App.js`
5. 注入 `<link href="ly-plugin://plugin/encoded/style.css">`，再通过 `<script src="...">` 加载 App.js
6. 协议处理器拦截请求，从 `plugins/<pluginDir>/` 读取文件并返回
7. App.js 执行后设置 `window.__LY_PLUGIN__`，加载器读取并 resolve 为 Vue 组件
8. 插件 UMD 使用全局 `Vue`、`ElementPlus`、`Icon`（由主程序在启动时暴露）

### 6.3 宿主环境暴露

主程序在 `main.js` 中通过 `exposePluginHost()` 设置：

```js
window.Vue = Vue
window.ElementPlus = ElementPlus
window.Icon = Icon
```

插件构建时使用这些作为 externals，不打包 Vue/ElementPlus/Icon。

## 7. 构建流程

### 7.1 本地开发

```bash
# 1. 在 plugins-ext 下创建插件目录
mkdir plugins-ext/我的插件

# 2. 写入 manifest.json 和 App.vue

# 3. 构建插件（生成 App.js 和 style.css）
npm run build:plugins              # 构建所有插件
npm run build:plugins -- 秀恩爱    # 仅构建指定插件，加快开发迭代

# 4. 启动主程序
npm run dev
```

修改插件源码后需手动执行 `npm run build:plugins` 重新构建。

### 7.2 构建产物

执行 `npm run build:plugins` 后：

1. 将 `plugins-ext/<插件名>/` 的源码构建并输出到 `plugins/<插件名>/`（含 manifest.json、App.js、style.css 等）
2. 为每个插件生成 `plugins/<插件名>.zip`，可直接用于插件市场分发

主程序开发时从 `plugins/` 目录加载插件。

## 8. 分发与安装

### 8.1 打包插件

插件市场分发的 zip 需包含：

- `manifest.json`
- `App.js`（必填，由 `build:plugins` 生成）
- `style.css`（可选，由 `build:plugins` 生成）
- 其他资源（如有）

**zip 目录结构建议**：顶层直接包含上述文件，或顶层为单目录（目录内包含上述文件）。安装时会解压到 `~/.ly/tools/plugins/<插件名>`。

### 8.2 插件名与 zip 文件名

插件目录名由 zip 文件名决定（去掉 `.zip` 后解码）。例如：

- `端口查杀.zip` → 目录名 `端口查杀`
- `port-killer.zip` → 目录名 `port-killer`

`manifest.json` 的 `id` 与目录名可不同，主程序通过 `id` 查找插件（如独立窗口）。

### 8.3 插件市场 index.json

插件市场通过 `index.json` 提供中文名称映射，格式示例：

```json
{
  "plugins": [
    { "filename": "端口查杀.zip", "title": "端口查杀" }
  ]
}
```

## 9. 特殊插件类型

### 9.1 全屏 + 独立窗口插件

锁屏、关灯、碎屏等插件通常需全屏覆盖，并打开独立 Electron 窗口。在 manifest 中声明 `fullScreen: true` 和 `electronAction`/`electronActions`：

```json
{
  "id": "lock-screen-light-off",
  "fullScreen": true,
  "electronAction": "openLightOffWindow"
}
```

点击卡片时，主程序会调用 `electronAPI[electronAction]()`；若需多个回调，可配置 `electronActions: ["openLightOffWindow", "closeLockWindow"]`。主程序不再硬编码插件 id，完全由 manifest 驱动。

部分插件（如锁屏、碎屏）使用独立 HTML 文件，主进程通过 `getPluginDirById(id)` 查找目录并加载：

- `lock-screen.html`（锁屏）
- `light-off.html`（关灯）
- `shatter-screen.html`（碎屏）

### 9.2 独立窗口加载（?pluginId=xxx）

支持以独立窗口打开的插件，URL 带 `?pluginId=xxx`，`pluginId` 与 `manifest.id` 一致，例如：

- `?pluginId=lock-screen-light-off` → 关灯
- `?pluginId=lobster` → 养龙虾（如有）

## 10. 常见问题

### Q: 插件显示「插件未构建」？

请执行 `npm run build:plugins` 生成 `App.js` 和 `style.css`。

### Q: 插件显示「插件加载失败」？

可能原因：
1. `App.js` 不存在，或 `manifest.json` 格式错误
2. **ly-plugin:// URL 格式问题**：若将 pluginDir（含中文）放在 URL 的 host 部分（如 `ly-plugin://%E7%AB%AF.../App.js`），部分环境下 `protocol.handle` 解析 hostname 时会出现异常导致 404。应使用 path 格式：
   - 正确：`ly-plugin://plugin/<encodeURIComponent(pluginDir)>/App.js`
   - 错误：`ly-plugin://<encodeURIComponent(pluginDir)>/App.js`

### Q: 打包后插件从哪加载？

打包后插件目录为 `~/.ly/tools/plugins`，用户从插件市场安装的 zip 会解压到此目录。

### Q: 如何调试插件？

插件与主程序共享同一渲染进程，可直接在 DevTools 中调试。独立窗口模式下，每个窗口有独立的 DevTools。
