## 1. 概述

插件是独立的 Vue 组件，通过 `manifest.json` 声明元数据，在运行时由主程序加载。主程序与插件解耦，插件需预构建为 UMD 格式的 `App.js`，并依赖主程序提供的 Vue、Element Plus、Icon。

## 2. 目录结构

```
plugins-ext/
└── 端口查杀/              # 插件目录名（建议与插件标题一致）
    ├── manifest.json     # 必填，插件元数据
    ├── App.vue           # 必填，插件入口组件
    ├── App.js            # 构建产物，由 build:plugins 生成
    └── style.css         # 构建产物，由 build:plugins 生成
```

### 2.1 插件目录与路径

| 环境 | 插件目录 |
|------|----------|
| 开发 | `./plugins`（由 `build:plugins` 从 `plugins-ext/` 构建输出） |
| 打包后 | `~/.ly/tools/plugins`（用户目录） |

源码在 `plugins-ext/`，执行 `npm run build:plugins` 会构建到 `plugins/` 并生成各插件的 zip 包；主程序从 `plugins/` 加载。

## 3. manifest.json

### 3.1 必填字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | string | 插件唯一标识，如 `port-killer` |
| `route` | string | 路由路径，如 `/port-killer` |
| `icon` | string | 图标，如 `ri:terminal-box-line`（Iconify 格式） |
| `description` | string | 插件描述 |
| `color` | string | 卡片渐变色，如 `linear-gradient(135deg, #667eea 0%, #764ba2 100%)` |
| `supportedOS` | string[] | 支持的系统，如 `["win64"]`、`["win64","linux"]` |

### 3.2 可选字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `title` | string | 显示名称，缺省时使用目录名 |
| `version` | string | 版本号 |
| `fullScreen` | boolean | 是否全屏显示 |
| `electronAction` | string | 点击卡片时调用的 electronAPI 方法名（如 `openLightOffWindow`） |
| `electronActions` | string[] | 点击卡片时依次调用的多个 electronAPI 方法名，可与 `electronAction` 共存 |
| `minLyToolsVersion` | string | 最低主程序版本 |

当 `fullScreen` 为 true 且配置了 `electronAction`/`electronActions` 时，主程序会在点击卡片时调用对应方法（若 electronAPI 中存在），否则走默认路由跳转。`electronAction` 先执行，再执行 `electronActions` 中各方法。

### 3.3 supportedOS 取值

- `win64` / `win32` / `windows` → Windows
- `linux` → Linux
- `darwin` / `macos` → macOS

### 3.4 完整示例（含所有属性）

```json
{
  "id": "my-plugin",
  "route": "/my-plugin",
  "title": "我的插件",
  "icon": "ri:terminal-box-line",
  "description": "插件功能描述，展示在首页卡片上",
  "color": "linear-gradient(135deg, #667eea 0%, #764ba2 100%)",
  "version": "1.0.0",
  "supportedOS": ["win32", "linux", "darwin"],
  "fullScreen": true,
  "electronAction": "openLightOffWindow",
  "electronActions": ["closeLockWindow"],
  "minLyToolsVersion": "1.0.0"
}
```

> `fullScreen` 为 `true` 且配置了 `electronAction`/`electronActions` 时，点击卡片会调用对应 electronAPI 方法；否则走默认路由跳转。普通插件只需省略 `fullScreen`、`electronAction`、`electronActions` 即可。

| 属性 | 必填 | 说明 |
|------|:----:|------|
| `id` | ✓ | 插件唯一标识，用于路由、独立窗口 `?pluginId=xxx` |
| `route` | ✓ | 路由路径，如 `/port-killer` |
| `icon` | ✓ | 图标（Iconify 格式，如 `ri:terminal-box-line`） |
| `description` | ✓ | 插件描述 |
| `color` | ✓ | 卡片渐变色 |
| `supportedOS` | ✓ | 支持系统：`win32`/`win64`/`windows`、`linux`、`darwin`/`macos` |
| `title` | | 显示名称，缺省时使用插件目录名 |
| `version` | | 版本号，展示在卡片上 |
| `fullScreen` | | 是否全屏，默认 `false` |
| `electronAction` | | 点击卡片时调用的 electronAPI 方法名 |
| `electronActions` | | 点击卡片时依次调用的多个 electronAPI 方法名 |
| `minLyToolsVersion` | | 最低主程序版本，不满足则插件不加载 |

**注意**：`pluginDir` 由主程序在加载时注入（插件目录名），无需在 manifest 中声明。

## 4. App.vue

### 4.1 基本结构

插件入口为单文件组件 `App.vue`，使用 Vue 3 Composition API、Element Plus、@iconify/vue。与主程序用法一致，无需额外配置。

```vue
<template>
  <div class="my-plugin">
    <h2>我的插件</h2>
    <!-- 使用 Element Plus 组件 -->
    <el-button type="primary" @click="handleClick">点击</el-button>
  </div>
</template>

<script setup>
import { ref } from 'vue'
import { ElMessage } from 'element-plus'
import { Icon } from '@iconify/vue'

const count = ref(0)
function handleClick() {
  count.value++
  ElMessage.success(`点击了 ${count.value} 次`)
}
</script>

<style scoped>
.my-plugin { padding: 20px; }
</style>
```

### 4.2 独立窗口模式

插件可通过 `?pluginId=xxx` 以独立窗口打开。若需区分主窗口和独立窗口，可接收 `standalone` 属性：

```vue
<script setup>
const props = defineProps({
  standalone: { type: Boolean, default: false }
})
// standalone 为 true 时表示在独立窗口运行
</script>
```

## 6. 运行时加载机制

### 6.1 自定义协议 ly-plugin://

主进程注册 `ly-plugin://` 协议，从插件目录提供静态文件。完整定义见 [ly-plugin协议定义.md](./ly-plugin协议定义.md)。

**URL 格式**：`ly-plugin://plugin/<encodeURIComponent(pluginDir)>/<file>`

| 示例 | 说明 |
|------|------|
| `ly-plugin://plugin/%E7%AB%AF%E5%8F%A3%E6%9F%A5%E6%9D%80/App.js` | 插件主入口（UMD） |
| `ly-plugin://plugin/%E7%AB%AF%E5%8F%A3%E6%9F%A5%E6%9D%80/style.css` | 插件样式 |

> **说明**：pluginDir 必须放在 path 中，避免 host 编码解析问题导致加载失败。

**协议处理流程**（主进程 `protocol.handle`）：

```
请求 URL: ly-plugin://plugin/<encoded>/App.js
    │
    ▼
解析 pathname → parts = [encoded, "App.js"]
    │
    ▼
pluginDir = decodeURIComponent(parts[0])
file = parts.slice(1).join("/")
    │
    ▼
filePath = pluginsRoot/pluginDir/file
    │
    ▼
校验：filePath 在 pluginsRoot 内 且 文件存在
    │
    ▼
读取文件 → 按扩展名设置 MIME → 返回 Response
```

**MIME 映射**：`.js` → `application/javascript`，`.css` → `text/css`，`.json` → `application/json`

### 6.2 加载流程

1. 主进程 `getPluginList()` 扫描插件目录，读取各插件的 `manifest.json`
2. 路由根据插件列表动态注册 `path: manifest.route`
3. 访问插件路由时，`getPluginComponentLoader(pluginDir)` 被调用
4. 渲染进程 IPC 调用 `getPluginEntryUrl(pluginDir)` 获取 `ly-plugin://plugin/encoded/App.js`
5. 注入 `<link href="ly-plugin://plugin/encoded/style.css">`，再通过 `<script src="...">` 加载 App.js
6. 协议处理器拦截请求，从 `plugins/<pluginDir>/` 读取文件并返回
7. App.js 执行后设置 `window.__LY_PLUGIN__`，加载器读取并 resolve 为 Vue 组件
8. 插件 UMD 使用全局 `Vue`、`ElementPlus`、`Icon`（由主程序在启动时暴露）

### 6.3 宿主环境暴露

主程序在 `main.js` 中通过 `exposePluginHost()` 设置：

```js
window.Vue = Vue
window.ElementPlus = ElementPlus
window.Icon = Icon
```

插件构建时使用这些作为 externals，不打包 Vue/ElementPlus/Icon。

## 7. 构建流程

### 7.1 本地开发

```bash
# 1. 在 plugins-ext 下创建插件目录
mkdir plugins-ext/我的插件

# 2. 写入 manifest.json 和 App.vue

# 3. 构建插件（生成 App.js 和 style.css）
npm run build:plugins              # 构建所有插件
npm run build:plugins -- 秀恩爱    # 仅构建指定插件，加快开发迭代

# 4. 启动主程序
npm run dev
```

修改插件源码后需手动执行 `npm run build:plugins` 重新构建。

### 7.2 构建产物

执行 `npm run build:plugins` 后：

1. 将 `plugins-ext/<插件名>/` 的源码构建并输出到 `plugins/<插件名>/`（含 manifest.json、App.js、style.css 等）
2. 为每个插件生成 `plugins/<插件名>.zip`，可直接用于插件市场分发

主程序开发时从 `plugins/` 目录加载插件。

## 8. 分发与安装

### 8.1 打包插件

插件市场分发的 zip 需包含：

- `manifest.json`
- `App.js`（必填，由 `build:plugins` 生成）
- `style.css`（可选，由 `build:plugins` 生成）
- 其他资源（如有）

**zip 目录结构建议**：顶层直接包含上述文件，或顶层为单目录（目录内包含上述文件）。安装时会解压到 `~/.ly/tools/plugins/<插件名>`。

### 8.2 插件名与 zip 文件名

插件目录名由 zip 文件名决定（去掉 `.zip` 后解码）。例如：

- `端口查杀.zip` → 目录名 `端口查杀`
- `port-killer.zip` → 目录名 `port-killer`

`manifest.json` 的 `id` 与目录名可不同，主程序通过 `id` 查找插件（如独立窗口）。

### 8.3 插件市场 index.json

插件市场通过 `index.json` 提供中文名称映射，格式示例：

```json
{
  "plugins": [
    { "filename": "端口查杀.zip", "title": "端口查杀" }
  ]
}
```

## 9. 特殊插件类型

### 9.1 全屏 + 独立窗口插件

锁屏、关灯、碎屏等插件通常需全屏覆盖，并打开独立 Electron 窗口。在 manifest 中声明 `fullScreen: true` 和 `electronAction`/`electronActions`：

```json
{
  "id": "lock-screen-light-off",
  "fullScreen": true,
  "electronAction": "openLightOffWindow"
}
```

点击卡片时，主程序会调用 `electronAPI[electronAction]()`；若需多个回调，可配置 `electronActions: ["openLightOffWindow", "closeLockWindow"]`。主程序不再硬编码插件 id，完全由 manifest 驱动。

部分插件（如锁屏、碎屏）使用独立 HTML 文件，主进程通过 `getPluginDirById(id)` 查找目录并加载：

- `lock-screen.html`（锁屏）
- `light-off.html`（关灯）
- `shatter-screen.html`（碎屏）

### 9.2 独立窗口加载（?pluginId=xxx）

支持以独立窗口打开的插件，URL 带 `?pluginId=xxx`，`pluginId` 与 `manifest.id` 一致，例如：

- `?pluginId=lock-screen-light-off` → 关灯
- `?pluginId=lobster` → 养龙虾（如有）

## 10. 常见问题

### Q: 插件显示「插件未构建」？

请执行 `npm run build:plugins` 生成 `App.js` 和 `style.css`。

### Q: 插件显示「插件加载失败」？

可能原因：
1. `App.js` 不存在，或 `manifest.json` 格式错误
2. **ly-plugin:// URL 格式问题**：若将 pluginDir（含中文）放在 URL 的 host 部分（如 `ly-plugin://%E7%AB%AF.../App.js`），部分环境下 `protocol.handle` 解析 hostname 时会出现异常导致 404。应使用 path 格式：
   - 正确：`ly-plugin://plugin/<encodeURIComponent(pluginDir)>/App.js`
   - 错误：`ly-plugin://<encodeURIComponent(pluginDir)>/App.js`

### Q: 打包后插件从哪加载？

打包后插件目录为 `~/.ly/tools/plugins`，用户从插件市场安装的 zip 会解压到此目录。

### Q: 如何调试插件？

插件与主程序共享同一渲染进程，可直接在 DevTools 中调试。独立窗口模式下，每个窗口有独立的 DevTools。
